The first proper week of practical work this semester started in the Object Oriented Programming lab, where I was assigned the task of building a complete Library Management System in C#. It was one of the most practical and challenging tasks I'd done so far in the course, because it required pulling together almost everything we'd learned and applying it inside a single project instead of isolated examples. The system needed to manage books, members, and borrowing records, so I started by identifying the main entities and giving each one its own class. The `Book` class held private fields like `BookID`, `Title`, `Author`, and `AvailableCopies`, with public properties built on getters and setters so I could control access and enforce validation — `AvailableCopies`, for instance, could never go below zero, which I enforced directly in the setter.

The `Member` class followed a similar pattern, with private fields for `MemberID`, `Name`, and `BorrowedBooks`. The part I found most useful, though, was writing `BorrowBook()` and `ReturnBook()`, which simulated the actual mechanics of lending — a book could only be borrowed if copies were available, and could only be returned if the member had genuinely borrowed it in the first place. That's where encapsulation stopped being an abstract term and started feeling like something with real consequences in the code. I extended the system further with a `PremiumMember` subclass that inherited from `Member` and was allowed to borrow more books at once, which gave me hands-on practice with the `base` keyword and constructor chaining. By the end of it, encapsulation and inheritance felt less like lecture slides and more like tools I'd actually used to solve a real design problem.

That same week, in Database Systems, I worked on something just as hands-on but in a completely different direction — applying two fundamental Relational Algebra operators, Selection and Projection, on real datasets. This lab mattered a lot because it closed the gap between the theory taught in class and what it actually looks like in SQL. The Selection operator, written as σ, filters rows based on a condition, and I used it on a STUDENT dataset to pull out students who'd scored above 80 marks, and separately on a BRANCH dataset to find HBL branches with total deposits above 500,000,000 PKR. Writing the relational algebra expression first and then converting it into a SQL `WHERE` clause made the connection between mathematical notation and actual query syntax click in a way that just reading about it never had.

The Projection operator, π, works differently — it selects specific columns while automatically removing duplicate rows. I practiced this by pulling just the `Name` and `Marks` columns from the student table, and `BranchCode`, `City`, and `Deposits` from the branch table. I also got some practice with the Rename operator (ρ), which is useful for giving cleaner names to result relations and their attributes. This course is taught by Dr. Bilal Ahmad, and the way he structures these sessions — theory first, then immediately into hands-on SQL — is a big part of why translating relational algebra into MySQL queries started feeling intuitive this early in the semester rather than something I'd struggle with later.

The third thing that stood out this week wasn't technical at all — it was a Communication Skills assignment where we had to create a vlog on the Physical Fitness Test for Armed Forces. It's a topic that matters to a lot of young Pakistanis who consider joining the Army, Navy, or Air Force, so I had real motivation to get it right. I covered the one-mile run, push-ups, sit-ups, pull-ups, and the medical and height-weight requirements, along with the importance of consistent training, diet, sleep, and mental preparation. Recording and editing it taught me more about verbal communication than I expected — things like body language, eye contact with the camera, and structuring content so it flows naturally from point to point. It was a good reminder that communication skills aren't just about writing; they're about how you speak, present, and actually connect with whoever's on the other side.

Looking back at this first week of real lab work, what stuck with me wasn't any single concept but how differently each course approached "doing" something practical — OOP through building a working system, Database Systems through bridging math and query syntax, and Communication Skills through stepping in front of a camera. All three ended up reinforcing the same idea: theory only really sets in once you're forced to apply it.
